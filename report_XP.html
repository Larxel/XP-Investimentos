Resumo

Iniciei esse desafio criando algumas features com base nos dados fornecidos. Apesar da abordagem clÃ¡ssica girar em torno de uma EDA, comecei criando um modelo baseline para entÃ£o focar minha atenÃ§Ã£o nas features mais importantes.

Como mÃ©trica chave, usei auc, uma vez que ela Ã© talvez a mÃ©trica mais flexÃ­vel de todas, permitindo Pela modelagem baseline, descobri problemas no dataset, como um data leak que causa grande perda em modelos.

Para sanar esse cenÃ¡rio, realizei uma EDA olhando feature Ã  feature, selecionando as tidas como mais relevantes. Como resultado da EDA, identifiquei o leak como sendo fonte da depedÃªncia entre clientes e a eliminei.

A exploraÃ§Ã£o do leak garante uma auc de 0.9786 no holdout set e dado que o test set possui os mesmo clientes contidos no leak, sua auc tambÃ©m tende a ser alta - mas pela invalidade prÃ¡tica desse mÃ©todo, eu o descartei.

Em seguida submeti um modelo duplamente validado com uma rodada de cross validation e outra em um holdout set estratificado. Por fim, uma combinaÃ§Ã£o linear de um Lightgbm com uma DNN feita em Keras geraram uma auc de 0.8886 usando apenas 8 features.

Modelagem SemiFinal e CorreÃ§Ã£o do Leak

Com as variÃ¡veis selecionadas e o leak identificado, Ã© hora de modelar! Isolando as features selecionadas, podemos entÃ£o fazer train/holdout split.

Removendo a duplicidade de registros de mesmo clientes diminuiremos drasticamente o tamanho do dataset, mas ao mesmo tempo removeremos o leak ao forÃ§ar o modelo Ã  compreender o fenÃ´meno, ao invÃ©s de ser contaminado por observaÃ§Ãµes interdependentes.

# [] Isolate selected features
selected_features = c('retorno','var38','time_dif','month_x','month_y','numday_x','numday_y','var1','categoria') #'grupo' removed for too high RAM cost
full.data = full.data %>% as.data.frame()
training_data = full.data[,selected_features]

# [] Train / Holdout Split
splitr = sample.split(training_data$retorno, SplitRatio = 0.7)
train  = subset(training_data, splitr == TRUE)
probe  = subset(training_data, splitr == FALSE)

# [] Leak removal
truthfull.data = train[!duplicated(train$ID),]
truthfull.data$ID = NULL

Em seguida, criaremos um randomforest tunado com cross validation, treinado sobre o training set sem leak (truthfull.data) e validado duas vezes.

O cenÃ¡rio de data leakage Ã© na realidade um dos mais preocupantes, na medida em que age na fonte de qualquer observaÃ§Ã£o e pode facilmente inviabilizar uma anÃ¡lise ou modelo. Por essa razÃ£o, usei a dupla validaÃ§Ã£o (cv + holdout) para garantir a integridade dos resultados.

Em relaÃ§Ã£o Ã¡ mÃ©trica escolhida, para aplicaÃ§Ãµes prÃ¡ticas a auc pode conciliar diferentes necessidades de thresholds, como em fraude (um threshold para aviso e outro para bloqueio do cartÃ£o), alÃ©m de naturalmente conduzir Ã  acurÃ¡cia. Sem um caso de uso ou entendimento do que Ã© o target, creio que essa seria a melhor mÃ©trica.

# [] Set CV Parameters and Grid
train.ctrl = trainControl(method="cv",number=4,classProbs=T,summaryFunction = twoClassSummary)
rf.grid    = expand.grid(mtry=c(3,4,5),
                         splitrule='gini',
                         min.node.size=1)

# [] Reencode targets (so caret won't crash)
train$retorno  = ifelse(train$retorno == 1,"Sim","Nao")
probe$retorno  = ifelse(probe$retorno == 1,"Sim","Nao")


# [] Train semifinal model (CV + Holdout validated)
rf.model = train(as.factor(retorno) ~ ., data=truthfull.data, method="ranger", num.trees=500,
                 trControl = train.ctrl, metric='ROC', tuneGrid = rf.grid) #0.83582
# [] Measure results
rf.pred = predict(rf.model, probe,type = 'prob')
table(probe$retorno, rf.pred$Sim >= 0.5)

probe$retorno  = ifelse(probe$retorno == "Sim",1,0)

glmnet::auc(probe$retorno, rf.pred$Sim) #0.833531
